BUG #01 — instanceof AppError falha no Error Handler global
─────────────────────────────────────────────────────────────────────────────
GRAVIDADE:  ALTA
ARQUIVO:    apps/api/src/app.ts (linhas 88-100)
STATUS:     PARCIALMENTE CORRIGIDO (workaround aplicado, não é fix definitivo)

DESCRIÇÃO:
  Em ESM (módulos ES nativos do Node 20), a checagem `instanceof` para
  classes lançadas de módulos diferentes pode falhar silenciosamente quando
  os protótipos não compartilham a mesma referência de cadeia.
  
  Isso fazia com que erros intencionais como "credenciais inválidas" (401)
  fossem engolidos e retornados como "500 Internal Server Error".

WORKAROUND APLICADO:
  Adicionamos `error.name === "AppError"` como checagem alternativa.
  Isso funciona, mas é frágil — se alguém renomear a classe, quebra.

CORREÇÃO DEFINITIVA RECOMENDADA:
  Usar um Symbol.for('AppError') ou um campo discriminante estático:
    static readonly brand = 'AppError' as const;
  E checar via: if ('brand' in error && error.brand === 'AppError')

BUG #02 — Múltiplas instâncias de PrismaClient (connection leak)
─────────────────────────────────────────────────────────────────────────────
GRAVIDADE:  ALTA
ARQUIVOS:   
  - apps/api/src/modules/auth/auth.service.ts:9
  - apps/api/src/modules/tasks/tasks.service.ts:6
  - apps/api/src/modules/dashboard/dashboard.service.ts:3
  - apps/api/src/modules/clients/clients.service.ts (mesma instância local)
  - apps/api/src/database/prisma/seed.ts:4
  (e possivelmente todos os demais services)

DESCRIÇÃO:
  Cada arquivo de service cria sua própria instância:
    const prisma = new PrismaClient();

  Isso gera N conexões separadas ao PostgreSQL, cada uma com seu próprio
  pool interno. Em produção, isso pode:
  - Esgotar os limites de conexão do PostgreSQL
  - Gerar warnings de "too many Prisma Clients" nos logs
  - Dificultar transações que precisem de consistência cross-service

CORREÇÃO RECOMENDADA:
  Criar um singleton centralizado:
    // apps/api/src/shared/database/prisma.ts
    import { PrismaClient } from "@prisma/client";
    export const prisma = new PrismaClient();

  E importar de lá em todos os services.

BUG #08 — accessToken armazenado no localStorage (via Zustand persist)
─────────────────────────────────────────────────────────────────────────────
GRAVIDADE:  SEGURANÇA — MÉDIA
ARQUIVO:    apps/web/src/stores/auth.store.ts:48-54

DESCRIÇÃO:
  O auth store persiste o accessToken e refreshToken diretamente no
  localStorage do navegador via `persist({ name: "satomiq-auth" })`.
  
  Isso os torna acessíveis a qualquer JavaScript rodando na mesma
  origin, incluindo scripts de terceiros ou extensões maliciosas.
  
  Em caso de XSS, tokens podem ser exfiltrados silenciosamente.

RECOMENDAÇÃO:
  - Mover refreshToken para httpOnly cookies (server-set)
  - Manter accessToken apenas em memória (sem persist)
  - Ou pelo menos, não persistir o refreshToken e aceitar re-login

INC #01 — Tipagem fraca generalizada no frontend (unknown abuse)
─────────────────────────────────────────────────────────────────────────────
GRAVIDADE:  MÉDIA
ARQUIVOS:   Todos os services e hooks em apps/web/src/services/*.ts

DESCRIÇÃO:
  Praticamente todos os services usam ApiResponse<unknown> ou
  ApiResponse<unknown[]> para tipar respostas da API:
    const { data } = await api.get<ApiResponse<unknown[]>>(`/clients?${params}`);

  Em seguida, nos componentes, fazem cast com `as`:
    const clients = (data?.data as ClientRecord[] | undefined) ?? [];

  Isso elimina toda a garantia de tipo do TypeScript. Se a API mudar
  a forma de um campo, o frontend não acusa nos tipos — simplesmente
  quebra em runtime.

CORREÇÃO RECOMENDADA:
  Criar interfaces de resposta compartilhadas no packages/shared e
  usá-las tanto no backend quanto no frontend.

INC #03 — Refresh token interceptor pode causar loop infinito
─────────────────────────────────────────────────────────────────────────────
GRAVIDADE:  MÉDIA
ARQUIVO:    apps/web/src/services/api.ts:22-56

DESCRIÇÃO:
  O interceptor de resposta captura erros 401 e tenta refresh.
  Se o próprio refresh retornar 401, ele chama logout() corretamente.
  
  Porém, se o refresh retornar um outro erro (ex: 500, timeout),
  o catch genérico faz logout() — o que pode ser agressivo demais.
  
  Adicionalmente, não há flag de "isRefreshing" para prevenir que
  múltiplas requisições simultâneas disparem N tentativas de refresh
  concorrentes.

CORREÇÃO RECOMENDADA:
  Implementar um mutex de refresh:
    let isRefreshing = false;
    let pendingRequests: Array<(token: string) => void> = [];

MOB #01 — Sidebar PERMANENTE sem toggle mobile — bloqueia toda a tela
─────────────────────────────────────────────────────────────────────────────
GRAVIDADE:  ██████████ CRÍTICA

ARQUIVO:    apps/web/src/layouts/AppLayout.tsx + Sidebar.tsx

DESCRIÇÃO:
  O AppLayout usa `flex h-screen` com a Sidebar como flex child direto.
  A sidebar tem largura mínima de 64px (colapsada) ou 260px (aberta).
  
  Em telas mobile (<640px), a sidebar AINDA OCUPA ESPAÇO horizontal,
  empurrando o conteúdo principal para uma área minúscula.
  
  NÃO EXISTE:
  - Nenhum media query para esconder a sidebar em mobile
  - Nenhum botão "hamburger" visível apenas em mobile
  - Nenhum comportamento de overlay/drawer para mobile
  - Nenhum mecanismo de swipe para abrir/fechar

  O botão de toggle (ChevronLeft) existe na sidebar expandida, mas
  ao colapsar, não há forma de reexpandir em mobile (o botão Menu
  no Topbar aparece somente quando sidebarCollapsed === true,
  e em telas estreitas ocupa espaço precioso).

IMPACTO:
  Em um celular, ~30-40% da largura é consumida pela sidebar, tornando
  o conteúdo ilegível e inutilizável. Esse é provavelmente o maior
  problema de UX mobile do sistema inteiro.

CORREÇÃO RECOMENDADA:
  - Detectar tamanhos de tela < 768px via hook useMediaQuery
  - Em mobile: sidebar como overlay/drawer absoluto com backdrop
  - Botão hamburger fixo no Topbar para abrir/fechar
  - Fechar sidebar ao clicar em qualquer NavItem (em mobile)
  - Transição suave com Framer Motion (já está importado)

╔══════════════════════════════════════════════════════════════════════════════╗
║  SEÇÃO 6 — PROBLEMAS MENORES E RECOMENDAÇÕES                               ║
╚══════════════════════════════════════════════════════════════════════════════╝

1. [CSS] Tailwind custom spacing sobrescreve as unidades padrão do Tailwind.
   O spacing no tailwind.config.ts usa valores absolutos (1: "4px", 2: "8px")
   em vez de múltiplos de rem. Isso pode gerar inconsistências se alguém
   usar classes como p-1 esperando 0.25rem mas receber 4px.

2. [ACESSIBILIDADE] Nenhum elemento interativo possui aria-label ou
   role="button". Todas as <button> sem texto acessível (ícone-only)
   são inacessíveis para screen readers.

3. [PERFORMANCE] O dashboard faz 18 queries no banco dentro de uma
   $transaction. Isso é eficiente em termos de consistência, mas pode
   ser lento à medida que as tabelas crescerem. Considerar materializar
   contadores ou usar views.

4. [FONT] O tailwind.config.ts referencia "Satoshi" e "General Sans" no
   fontFamily.display, mas não há importação dessas fontes no index.html
   ou no CSS. Essas fontes nunca carregam — o fallback para Inter é usado.

5. [SEO] As rotas são renderizadas client-side (SPA com React Router).
   Não há server-side rendering, meta tags dinâmicas, ou sitemap.
   Para um sistema interno isso é aceitável, mas SE for exposto
   publicamente, SEO será zero.

6. [DOCKER] O web Dockerfile (se existir) precisa ser verificado
   separadamente. O docker-compose.yml referencia apps/web/Dockerfile.

7. [ENV] A variável CORS_ORIGIN aceita múltiplos origins separados por
   vírgula (apps/api/src/app.ts:29), mas o valor default é uma string única.
   Se configurado incorretamente, CORS blocking pode ocorrer silenciosamente.

8. [SEGURANÇA] Não há rate limiting por IP no nível de container/reverse proxy.
   O rate limiter do Express usa memória in-process, que reseta a cada restart.
   Em produção, usar Redis ou similar para rate limiting persistente.
