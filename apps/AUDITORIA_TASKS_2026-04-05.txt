║  BUG #9 — BACKEND: updateTask() não permite remover assignee              ║
╚══════════════════════════════════════════════════════════════════════════════╝

GRAVIDADE:    BAIXA — limitação funcional
ARQUIVO:      apps/api/src/modules/tasks/tasks.service.ts — linhas 127
TIPO:         Lógica de update incompleta

CÓDIGO ATUAL (linha 127):
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ...(input.assigneeId !== undefined && { assigneeId: input.assigneeId })│
  └─────────────────────────────────────────────────────────────────────────┘

PROBLEMA:
  Para remover o assignee de uma tarefa (desatribuir), o cliente precisaria
  enviar `assigneeId: null`. Mas a interface UpdateTaskInput estende
  Partial<CreateTaskInput> onde assigneeId é `string | undefined`.
  `null` não é aceito pelo tipo.

  Se o frontend tentar enviar `assigneeId: null`, o TypeScript vai reclamar.
  Se enviar `assigneeId: undefined`, a condição `!== undefined` é falsa,
  então o campo não é atualizado — o assignee nunca é removido.

CORREÇÃO RECOMENDADA:
  Ajustar a interface UpdateTaskInput para aceitar null:
    assigneeId?: string | null;

  E no service, tratar explicitamente:
    ...(input.assigneeId !== undefined && { assigneeId: input.assigneeId })
  Isso já funciona se null for aceito como valor (Prisma aceita null para
  desconectar a relação opcional).

║  BUG #10 — BACKEND: isRefreshing mutex existe mas é module-scope          ║
╚══════════════════════════════════════════════════════════════════════════════╝

NOTA: Este bug afeta toda a autenticação e foi identificado na outra auditoria,
mas é relevante mencionar aqui pois impacta diretamente o funcionamento do
módulo Tasks quando o accessToken expira mid-session.

GRAVIDADE:    MÉDIA — pode causar logout inesperado ao usar Tasks
ARQUIVO:      apps/web/src/services/api.ts — linhas 22-81
TIPO:         Race condition no refresh token

PROBLEMA:
  O mutex `isRefreshing` e `pendingRequests` EXISTEM no código (linhas 22-28),
  e o interceptor parece implementado corretamente.

  PORÉM: Ao recarregar a página, o accessToken NÃO é restaurado do storage
  (auth.store.ts, linha 52-54: `partialize` só persiste user e isAuthenticated,
  NÃO persiste tokens). Isso significa que após reload:

  1. isAuthenticated = true (persiste)
  2. accessToken = null (não persiste)
  3. refreshToken = null (não persiste)

  O interceptor tenta refresh → não tem refreshToken → chama logout().
  O usuário é deslogado a cada reload de página.

  Isso impacta diretamente o Tasks: se o usuário navegar para /tasks
  após recarregar a página, será imediatamente redirecionado para /login.

CORREÇÃO RECOMENDADA:
  Ou persistir os tokens (com as ressalvas de segurança do BUG #08
  da auditoria anterior), ou implementar um mecanismo de "silent refresh"
  na inicialização do app que recupere os tokens via cookie httpOnly.
